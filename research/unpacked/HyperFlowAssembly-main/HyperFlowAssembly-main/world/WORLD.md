# World: GTA Finance Assembly

A markdown-first assembly world for building SessionMint / Tianezha as a sovereign session-state kernel with multichain identity, flow-governance, useful-compute markets, public and private superbrain routing, Windows-only desktop embodiment, browser/mobile finance games, Hyperliquid convergence, AI media surfaces, Conway-backed sub-agents, and bounded machine outputs.

This world follows Interface Assembly:
- new work is proposed first
- one sovereign owner exists per block
- shared contracts stay neutral as interfaces
- adapters remain edges, not state owners
- cross-boundary change requires owner-block participation
- Archivist records accepted truth
- PublicReadiness gates publication

## Foundational rules
1. One identity cluster per wallet.
2. Three-wallet identity is mandatory for primary governance participation.
3. Solana wallet = YES rail.
4. BNB wallet = NO rail.
5. Base wallet = delegate to Tianezha.
6. Votes count only when tokens are actually sent.
7. The public Tianezha router is the only coordination path between PrivateSuperBrainBox and sub-agents.
8. PrivateSuperBrainBox is backend-only.
9. PrivateSuperBrainBox may be deployed on Google Cloud by default and optionally on Conway, but its trust rules do not change.
10. DesktopWorldBox is Windows only.
11. Browser/mobile/chatbot surfaces do not run the VTuber LLM.
12. Users and cockpit operators may supply their own approved model providers and API keys under local-first sovereignty.
13. MYTHIV42 agents are Conway only.
14. Public/world/community agents may use the cheapest approved runtime.
15. Adapters are edges, not state owners.

## Human-readable summary
This system is not a launchpad. A token launches on its native ecosystem first, then its creator or community can pay a world activation fee to provision a world, Conway-backed services, a public brain, optional private services, compute markets, media, and game/social surfaces. Worlds can also be created without a token.

## Operator flow
1. Read this world file.
2. Read the clause for the box you want to touch.
3. Write or update a proposal in `world/proposals/`.
4. Scope the change inside one owner boundary.
5. Route cross-boundary changes through amendment or conflict flow.
6. Update registries.
7. Snapshot accepted truth in `world/merkle/current.md`.
8. Run public-readiness review before publishing.
